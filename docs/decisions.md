# Реестр решений

Каждое нетривиальное решение по проекту — сюда. Дата · решение · аргументация.

---

## 2026-05-12

### D-001: Стек
**Решение:** Next.js 16 (App Router) + TypeScript + Tailwind v4 + Framer Motion 12.
**Почему:** актуальный production-стек, поддержка mobile-first, Server Components, минимум JS на клиенте.

### D-002: Дизайн-система
**Решение:** Humble — Precise White Lab.
**Почему:** уже зафиксирована как корпоративная для Антона. Bricolage Grotesque + Geist, Canvas White, Electric Orange акцент, мягкие тени.

### D-003: Информационная архитектура
**Решение:** одностраничник с длинной прокруткой и якорями.
**Почему:** соответствует принципу простоты, mobile-first прокрутка естественна, легче поддерживать CMS на одной странице. Многостраничник — Этап 2 масштабирования.

### D-004: Порядок секций
**Решение:** Hero → Проекты-быстрый-обзор → Сторителлинг → Медиа → Коммерческие → Социальные → Социальный капитал → Партнёрства → Контакт.
**Почему:** Антон отдельно подчеркнул — сначала фото и проекты, чтобы знакомому не приходилось скроллить сторителлинг. Медиа сразу после потому что это точка входа большой части аудитории.

### D-005: Категоризация проектов
**Решение:** две категории — Коммерческие и Социальные/Благотворительные.
**Почему:** масштабируется, обе категории будут расти. Антон явно попросил такое разделение.

### D-006: Mobile-first
**Решение:** дизайн от 375px, потом масштабируем.
**Почему:** 90% трафика с мобильного по словам Антона.

### D-007: CMS
**Решение:** Sanity (предварительно).
**Почему:** лучший UX редактора для не-программиста (Ксеня Пиён), real-time preview, бесплатный план до 3 редакторов и 10K документов, multi-locale ready (для будущей англо-версии).
**Альтернативы рассматривались:** TinaCMS, Decap, Strapi, Notion API.
**Финальное решение:** ждёт ответа в ТЗ v1, вопрос 15.4.

### D-008: Бронь встреч
**Решение:** Cal.com (предварительно).
**Почему:** бесплатно, полная стилизация под Humble, мультитипы встреч (30/60/90), embed на сайт.
**Альтернативы рассматривались:** Calendly, SavvyCal.
**Финальное решение:** ждёт ответа в ТЗ v1, вопрос 15.3.

### D-009: Хостинг
**Решение:** Vercel (Hobby plan).
**Почему:** бесплатный, нативная поддержка Next.js, мгновенный deploy, привязка домена за минуты.

### D-010: Аналитика
**Решение:** Яндекс.Метрика.
**Почему:** так попросил Антон. Бесплатна, дает heatmap, российская юрисдикция.

### D-011: Локация проекта
**Решение:** `~/Claude3/projects/_shared/anton-site/`.
**Почему:** проект отдельный, с кодом + ассетами + документами. Под `projects/_shared/` потому что не строго личное, не строго бизнес — самостоятельная единица. По указанию Антона.

### D-012: Структура Google Drive
**Решение:** новая папка `12_Сайт` внутри Oreshkin Business, 6 подпапок (Документы / Контент / Фото / Лого / Видео / Скриншоты).
**Почему:** изолированное место под все материалы сайта, не смешивается с другими проектами.

### D-028: Фото-стратегия — прототип-first
**Решение:** собираем сначала прототип на placeholder-фото (один тестовый портрет + размытые силуэты в остальных слотах). После апрува композиции — выдаю точный фото-бриф под каждый слот, Антон присылает.
**Почему:** композиция диктует фото, не наоборот. Параллельно работаем — пока я кодирую, Антон не торопится с фото.

### D-029: Cal.com — настраиваю самостоятельно
**Решение:** настраиваю Cal.com бесплатный план самостоятельно с user-friendly интерфейсом. Если UX не подойдёт — переключаемся на Calendly Basic (бесплатный, неограниченное число встреч одного типа).
**Резервный план:** Calendly Basic как fallback.
**Слоты и длительности встреч:** Антон даст список позже — пока встраиваем embed-плейсхолдер.

### D-030: Подход к работе — максимальная автономность Антона
**Решение:** я веду все этапы самостоятельно (план → код → контент-драфты → прототип → подбор фото → деплой). Антон даёт обратную связь по присланным артефактам.
**Точки касания:** короткие mp4-кружки прогресса в Telegram + Google Doc на каждый этап. Антон не пишет сам — комментирует мою работу.
