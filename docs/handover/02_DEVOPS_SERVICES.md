# 02. DevOps · Сервисы, доступы, бюджет, операционка

Документ ревизии инфраструктуры сайта-визитки `anton-site` (`anton-site-omega.vercel.app`). Готовится под двойную цель: handover Ксении Парфёновой (`@xeniapm`, БА Антона) и развитие сайта на 6-12 месяцев.

Дата: 2026-05-17. Автор: DevOps Advisor (Claude). Контекст проекта: `HANDOFF.md`, `CLAUDE.md`, `docs/decisions.md`, `docs/audit-2026-05-14/MORNING_REPORT.html`, memory `project_anton_site`.

Принципы документа:
- Каждый сервис — реальный URL, реальная цена в рублях и долларах, конкретный следующий шаг.
- Где сервис создан — указан владелец и email-логин. Где не создан — оценка, кто создаёт, сколько занимает.
- Все российские сервисы — основные. Зарубежные — там, где альтернатива хуже по UX или невозможна (Vercel, Sanity, Cal.com, GitHub).
- Платёжная модель: всё, что списывается с карты, остаётся на Антоне. Ксении передаются только операционные доступы.

## ЧАСТЬ B. Матрица доступов для Ксении Парфёновой

Принципы:
- Ксения получает Editor/Member-роль везде, где может работать с контентом.
- Billing accounts, payment methods, root-доступ к Vercel и доменам — остаются за Антоном.
- На уровне scope: GitHub-доступ ограничен одним репо, Vercel-доступ ограничен одним проектом, Sanity-доступ — одним dataset.
- Все приглашения отправляются на её рабочий email (Ксения должна сообщить какой использует — рекомендация завести `xenia@antonoreshkin.ru` когда поднимем Yandex 360).

### Vercel

| Параметр | Значение |
|---|---|
| Роль для Ксении | Member (а не Owner) |
| Как добавить | https://vercel.com/team_VK9rtO4WoqN9i0PbXZh41Nva/settings/members → Invite Member → email Ксении → Role: Member |

## ЧАСТЬ E. Operations и мониторинг

### Алерты и health checks

Что отслеживаем (в порядке критичности):

1. **Uptime** (доступность сайта). Источник: UptimeRobot, проверка раз в 5 мин. Алерт при двух подряд failed checks (10 мин downtime). Адресат: Telegram-бот ошибок.
2. **Build failures** (Vercel). Источник: Vercel webhook → API route → Telegram. Алерт мгновенно при failed deploy.
3. **JS errors в проде** (Sentry). Алерт при >5 одинаковых ошибок за 5 минут. Telegram.
4. **Core Web Vitals** (LCP, CLS, INP). Источник: Vercel Analytics + Яндекс.Метрика. Алерт раз в неделю в Telegram-бот: «LCP вырос на 20%».
5. **Sanity API quota**. Источник: Sanity manage dashboard. Раз в неделю — проверка через API, алерт если >80% от 100k.
6. **Cal.com bookings**. Просто мониторинг роста через UI, не алерт.

Куда слать: **единый чат `сайт — алерты`** с ботом, в нём Антон + Ксения. Разные сервисы — разные эмодзи-маркеры в начале сообщения (без эмодзи в самом тексте). Например: 🟥 для downtime, 🟧 для build fail, 🟨 для Sentry, 🟩 для health-recovery.

Уточнение: пользователь предпочитает без эмодзи. Используем текстовые маркеры в `[BRACKETS]`: `[DOWN]`, `[BUILD-FAIL]`, `[ERROR]`, `[OK]`.

### Backup стратегия

Три уровня:

1. **Sanity dataset** (контент): ежедневный `sanity dataset export production` → загрузка в Backblaze B2 (private bucket).
2. **GitHub mirror** (код): дополнительный `git clone --mirror` → B2 раз в день. Защита от ситуации «GitHub удалил репо за политику».
3. **Google Drive 12_Сайт** (фото/видео-сырьё): уже автоматически синхронизируется с локальным Google Drive Антона. Дополнительно — раз в месяц `rclone copy` в B2.

Восстановление:
- Sanity: `sanity dataset import latest.tar.gz`.
- GitHub: `git clone --mirror backup.bundle ./anton-site`.
- Google Drive: ручное копирование.

Тест восстановления — раз в квартал на staging-окружение Vercel (отдельный проект `anton-site-restore-test`). Записать процедуру в Notion.

### Disaster Recovery

Сценарии и митигации:

| Сценарий | Что делаем | RTO (время восстановления) |

- На уровне Ксении: она видит PR в GitHub, может комментировать, но не мержит.
- Merge: squash-and-merge с осмысленным commit message в формате `feat: добавил блок XXX` / `fix: ...` / `chore: ...`.

### Как Ксения поднимает вопрос или блок

- **Канал**: Telegram, личный чат с Антоном или групповой «Сайт — алерты».
- **Формат сообщения**:
  - Что заметила (1-2 предложения).
  - Что попробовала (1 предложение).
  - Что нужно (от Антона или от Claude).
  - Срочность: blocker / important / nice-to-have.
- **SLA ответа**:
  - Blocker (сайт лёг или критичная ошибка) — Антон в течение 2 часов.
  - Important (контент или функционал) — 24 часа.
  - Nice-to-have — на еженедельном review.

### Документация изменений

- Каждый PR пишет sentence в `CHANGELOG.md` (создадим): «2026-06-XX — добавили блок XXX, переписали Hero, исправили баг с формой».
- Release notes — после крупных вех, в Google Doc «Журнал релизов сайта».
- Ксения видит ленту изменений в GitHub commits на main.

### Регулярные точки

- **Weekly review** (15 мин, четверг 17:00 MSK): Антон + Ксения по видео. Что было за неделю, что план на следующую, что блокирует.
- **Monthly site-review** (30 мин, первый понедельник месяца): Антон + Ксения. Аналитика (что работает, что не работает), приоритеты на месяц.
- **Quarterly tech-review** (1 час, раз в 3 мес.): Антон + Claude. Стек, зависимости, безопасность, рост нагрузки, бюджет.

## Заключение

Текущий стек на 2026-05-17 — минимально жизнеспособный: код на ноуте, деплой через Vercel, фото в Google Drive. Доменом, CMS, аналитикой и алертами пока не подключены.

Базовая зрелая конфигурация на ближайший квартал — около 200 ₽/год в обязательной части (домен), плюс 4000-5500 ₽/год на почту и резерв credentials manager. Все остальные сервисы — бесплатные планы их вендоров.

Стратегия handover Ксении: Editor-роли везде, где есть контент или операционка; Read-only там, где критичная инфраструктура; полное исключение из биллинга и domain ownership.

Главный месячный ритм после полного запуска: еженедельный 15-минутный sync Ксения-Антон, ежемесячный 30-минутный обзор аналитики, ежеквартальный технический review. Этого достаточно, чтобы сайт жил и развивался без постоянного включения Антона.
