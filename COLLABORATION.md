# Совместная разработка anton-site

Этот файл описывает как Антон Орешкин (`@antonoreshkin`) и Ксения Парфёнова (`@xeniapm`) работают с сайтом параллельно, в том числе с помощью Claude Code.

## Кратко

- Репозиторий: https://github.com/antonoreshkin/anton-site (приватный)
- Прод: https://anton-site-omega.vercel.app (Vercel project `anton-site`)
- Боевая ветка: `main` — защищена, прямой push запрещён
- Все правки идут через feature-ветки и Pull Request
- Vercel автоматически деплоит preview на каждый PR и прод на каждый merge в `main`

## Workflow для правок

### 1. Создать feature-ветку

```bash
git checkout -b feat/<коротко-что-делаем>
# примеры:
#   feat/zamenit-foto-vending
#   feat/dobavit-proekt-podcast
#   fix/sitemap-broken-links
#   content/sentyabr-2026
```

Префиксы:
- `feat/` — новая фича или раздел
- `fix/` — багфикс
- `content/` — правка только текстов/фото без кода
- `chore/` — техническое (зависимости, конфиги)

### 2. Сделать правки

Локально через Claude или вручную. Маленькими атомарными изменениями.

### 3. Закоммитить

```bash
git add -A
git commit -m "что и зачем (1 предложение)"
```

Заголовок коммита — на русском, активным глаголом, не более 72 символов.
Хорошо: `обновил фото вендинга на реальный кадр с аппаратом`
Плохо: `update`, `fixes`, `wip`

### 4. Push и открыть PR

```bash
git push -u origin feat/<имя-ветки>
gh pr create --title "<что меняем>" --body "<зачем + чек-лист>"
```

PR-описание (шаблон):
```markdown
## Что меняем
- бул лист изменений

## Зачем
- кратко мотивация

## Чек-лист
- [ ] Проверил локально (`npm run dev` + iPhone-режим в Chrome)
- [ ] Проверил голос и табу (CLAUDE.md → раздел «Стиль и табу»)
- [ ] Длинных тире нет
- [ ] АССИСТ+ заглавными
- [ ] Vercel preview deploy зелёный
```

### 5. Ревью (опциональное)

- Антон ревьюит **только если** Ксюша сама помечает PR тегом `needs-anton-review` или ставит его в reviewer
- По умолчанию — Ксюша мерджит свои PR сама, без ожидания
- Антон смотрит итог через прод-сайт и Vercel preview-комменты в PR
- Эскалация к Антону нужна когда: меняется позиционирование, добавляется новый проект на главную, правка может нарушить табу (Третьяков, ИРИ, Атланты, Signal TV, доли, суммы)

### 6. Merge

Squash merge. Удалить ветку после merge.

```bash
gh pr merge --squash --delete-branch
```

### 7. Vercel автоматом задеплоит на прод (~2 мин)

Проверить: https://anton-site-omega.vercel.app

## Если правка только текстовая (через Sanity CMS)

Когда Sanity подключим — мелкие текстовые правки идут через CMS, без PR:
- Ксюша / Антон правит в Sanity Studio
- Жмёт Publish
- Vercel ISR обновляет страницу на следующем запросе (или через webhook сразу)

PR-flow остаётся для правок кода, структуры, новых страниц, обложек, дизайна.

## Если конфликт веток

```bash
git checkout feat/моя-ветка
git fetch origin
git rebase origin/main
# разрешаем конфликты
git rebase --continue
git push --force-with-lease
```

Никогда `git push --force` без `--with-lease`.

## Claude в обеих сессиях

У Антона и у Ксюши Claude Code читает один и тот же `CLAUDE.md` в репо. Это гарантирует одинаковый стиль и одинаковые табу.

Если Claude меняет код в feature-ветке — Антон или Ксюша делает PR от своего имени (Claude не открывает PR напрямую от чужого аккаунта).

## Что нельзя

- Push прямо в `main` (защищено)
- Force push в `main`
- Коммитить секреты (.env, ключи, токены)
- Коммитить `node_modules/`, `.next/`, `.vercel/` (в `.gitignore`)
- Менять `package.json` без согласования (риск сломать билд)
- Удалять `.vercel/project.json` (привязка к Vercel project)
